# Skill: Scheduled Views for Search Acceleration

## Intent

Use Sumo Logic Scheduled Views to pre-aggregate data, dramatically accelerate dashboard panels and reporting queries, and cache expensive compute operations (threat intelligence, geo-IP lookups) for fast reuse over long time ranges.

## Prerequisites

- Administrator or power user access to create Scheduled Views
- A working aggregate log search query to use as the view definition
- Understanding of which queries are slow or expensive (see `search-performance-best-practices`)

## Context

**Use this skill when:**

- A dashboard panel or report is slow to load over multi-day or multi-week time ranges
- The same complex, compute-heavy operation (e.g., threat intel lookups, geo-IP) is repeated on every query
- An aggregate query needs to scale to weeks or months of data
- Queries against raw data are hitting scan or row-count limits in large environments

**Don't use this when:**

- The query shape changes frequently — views are fixed at definition time
- The time range is short (under a few hours) — raw data is fast enough
- You need fully flexible ad-hoc exploration — views have a fixed schema

---

## What Are Scheduled Views?

A Scheduled View is a **pre-aggregated data set** generated by running a query on a recurring schedule (every 1 minute by default) and storing the results back into Sumo Logic as a new indexed data set.

Queries run against a Scheduled View are dramatically faster because:
- The raw event count is massively reduced (e.g., millions of events → thousands of rows)
- Expensive operations (parse regex multi, threat intel lookups, geo-IP) are computed once at write time, not at every query
- The view has a fixed, indexed schema — queries scan the compact view data, not raw logs

**Potential speedup:** 10x to 100x or more, depending on the ratio of view rows to raw event rows.

---

## The "Base Camp" Mental Model

Climbing a mountain (large time-range aggregate query) in one go is hard. Scheduled Views are like base camps — you cache the work done on each stage so subsequent climbs start from a higher point.

```
Raw data               → 10s of millions of rows/day
↓ 1-minute view        → 1,440 rows/facet/day (one per minute)
↓ 1-hour view          → 24 rows/facet/day
↓ 1-day view           → 1 row/facet/day
↓ Weekly view          → 1 row/facet/week
```

Advanced users can build **multi-layer view architectures** using `save to index` scheduled searches that summarise 1m views into 1h views, then 1h views into 1d views — enabling extremely fast reporting over months of data.

## View Pattern 1: Aggregate Reporting

**Use case:** Dashboard panels showing time series or categorical breakdowns over large data sets (e.g., 7-day or 30-day web server status code trends).

### View Definition (runs every 1 minute, stores results to `apache_status`)

```
_sourceCategory=Labs/Apache/Access
| parse "HTTP/1.1\" * " as status_code
| timeslice 1m
| count by status_code, _timeslice
```

Stored as a Scheduled View named `apache_status`. The view has **one row per status code per minute** — dramatically smaller than raw event data (which could be millions of events per minute).

### Querying the View

```
// Fast time series panel for a 7-day dashboard:
_view=apache_status
| timeslice 1h
| sum(_count) as requests by status_code, _timeslice
| transpose row _timeslice column status_code
```

**Key:** Use `sum(_count)` — not `| count` — because the view already contains pre-aggregated counts. Using `| count` on a view counts view *rows*, not original events.

```
// Faster categorical summary:
_view=apache_status
| timeslice 1h
| sum(_count) as requests by status_code
| sort requests desc
```

**Result:** A panel that took minutes to load from raw data now loads in seconds from the view.

## View Pattern 2: Caching Heavy Compute Work

**Use case:** Queries that involve expensive operations — `parse regex multi`, threat intel lookups (`threatip`), geo-IP resolution (`geoip`), ASN lookups — that are slow and costly to rerun on every dashboard refresh.

### The Problem

Threat intelligence queries against raw WAF logs:
- Run `parse regex multi` on every event (expensive)
- Call `threatip` on every public IP (expensive, slow)
- Call `geoip` and `lookup asn` on each match (more compute)
- Must be re-run entirely on each query — no caching of intermediate results
- Historical threat intel state is lost as the lookups reflect current state, not state at the time of the event

### The Solution: Pre-Compute and Cache Threat Matches in a View

Store only the **positive matches** with all enrichment fields computed at write time.

**View Definition for `threat_geo_asn_aws_waf_v1`:**

```
_sourceCategory=Labs/AWS/WAF
| json field=_raw "httpRequest.clientIp" as src_ip
| where ispublicip(src_ip)
| json field=_raw "action"
| threatip src_ip
| where !(isempty(malicious_confidence))   // Only keep positive threat matches
| timeslice 1m
| "AWS" as vendor | "WAF" as product
| malicious_confidence as threat
| json field=raw_threat "threat_types"
| count by _timeslice, vendor, product, _sourcecategory, _source,
           src_ip, action, threat, actor, threat_types
| lookup asn, organization from asn://default on ip=src_ip
| geoip src_ip
| fields -latitude, longitude, country_name, state   // drop if not needed in view
```

This view stores only events where a threat was detected — typically a tiny fraction of total WAF log volume.

### Querying the Threat View

```
// 30-day threat trend by actor — loads in seconds:
_view=threat_geo_asn_aws_waf_v1
| timeslice 1d
| sum(_count) as events by actor, _timeslice
| transpose row _timeslice column actor
```

**Additional benefits of caching in a view:**
- **Historical forensic state preserved:** geo-IP and threat intel values are captured at the time of the event, not the time of the query (threat intel feeds change over time)
- **New use cases enabled:** the view can be used as a sub-query source for join operations, anomaly detection, or cross-source correlation
- **Faster by 100x+:** for 30-day panels, the original raw query might time out or take hours; the view loads in seconds

## View Schema Rules

When querying a Scheduled View, you must understand its schema:

1. **Check `reduceOnlyFields`:** The view exposes specific output columns. Use these names, not guesses.
2. **Use `sum()` not `count`:** The `_count` column in a view contains pre-aggregated event counts. To re-aggregate over time, use `sum(_count)`.
3. **Scope is `_view=<name>`:** Use `_view=` prefix, not `_index=` or `_sourceCategory=`.
4. **Use version patterns:** Versioned views are common (`apache_status_v1`, `apache_status_v2`). Query with `_view=apache_status_v*` to hit all versions.

## Creating a Scheduled View

1. Navigate to **Manage Data → Logs → Scheduled Views**
2. Click **+ Add** to create a new view
3. Enter a view name (lowercase, descriptive, short)
4. Paste the view definition query
5. Set the **start time** (data before this time will not be in the view)
6. Set the **retention period**
7. The view runs every 1 minute, appending the latest time window's results

**Alternatively:** A "save to index" scheduled search runs on a configurable schedule (every 15 minutes, hourly, daily) and writes its results to a named index. This is useful for coarser time-granularity views or multi-layer view architectures.

---

## Multi-Layer View Architecture

For very large organisations or long time-range reporting:

```
1m view  →  pre-aggregated at 1m granularity (1440 rows/facet/day)
   ↓ save-to-index scheduled search (hourly)
1h view  →  summarises 1m view (24 rows/facet/day)
   ↓ save-to-index scheduled search (daily)
1d view  →  summarises 1h view (1 row/facet/day)
```

Each layer is much smaller than the one below it. A 90-day query against the 1d view touches only 90 rows per facet — versus millions of raw events.
